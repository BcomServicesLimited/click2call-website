6. Setup Process (Step by Step)

Recommended Setup: Small Business with Reception Call Flow

For businesses wanting a main number that rings reception first, then overflows to other staff, we recommend:

  • 1 × Inbound Business Number ($10/mo ex GST) as the primary inbound line
  • 1 × Cloud PBX User per staff member who needs to make/receive calls ($25/mo ex GST each)

This setup allows group mailboxes, after-hours routing, and ring groups to be configured on the main line independently of individual user extensions. It provides greater flexibility for call flow programming and future scalability – staff can be added or removed from the call flow without changing the main number's configuration.

Step-by-Step Setup Process

From the moment a customer signs up to the moment their first call comes in, here is exactly what happens:

  1. Create Your Account: Complete the signup form at portal.click2call.com.au/join. The form takes around five minutes. The account will be activated within 24 hours.
  2. Add a Temporary Number: Once active, add a temporary local number from the portal so you can start building the system immediately.
  3. Set Up Your Users: Add each member of the team as a user in the portal. Configure call routing, ring groups, or auto-attendant menus.
  4. Install the Apps: Download and install the Click2Call softphone on each team member's PC or configure physical handsets.
  5. Test the System: Make test calls to and from the temporary number to ensure all routing and apps are working correctly.
  6. Divert Your Old Number: Set up a call diversion on the old phone system, pointing the main business number to the new temporary Click2Call number. The customer is now live on the new system.
  7. Port Your Number: Submit a port request to move the main number to Click2Call permanently. This takes 5–10 business days. Because of the diversion, there is zero downtime.
  8. Remove Temporary Number: Once the port completes, the main number is officially on Click2Call. The customer can cancel their old service and remove the temporary number from their account.

12. Microsoft Teams Calling (Direct Routing)

12.1 Product Overview and Architecture

Click2Call Microsoft Teams Calling enables businesses to make and receive external phone calls directly within the Microsoft Teams application. The service uses Direct Routing architecture, which is a Microsoft-approved method for connecting Teams to the public switched telephone network (PSTN).

Unlike basic SIP trunk providers that only supply a phone number and a connection, Click2Call integrates a full Cloud PBX beneath the Teams interface. This architectural difference means customers receive enterprise-grade PBX features natively, without needing third-party add-ons or complex integrations. The core value proposition is providing two systems for one price: customers get the familiar Microsoft Teams interface powered by Click2Call's advanced Cloud PBX infrastructure.

12.5 Infrastructure, Data Residency, and Support

Data Residency and Compliance

All call recordings, transcripts, AI-generated outputs, and account data are stored exclusively on infrastructure located in Australia and New Zealand. Click2Call does not use offshore servers for data storage or processing. This ensures compliance with the Privacy Act 1988 (Cth) and the Australian Privacy Principles, which is critical for businesses in regulated industries such as healthcare, finance, legal, and government.

Network and Support

The telecommunications network is based in Australia and New Zealand. Technical support is provided by a local Australian team.

Service Availability and Continuity

  • Available anywhere in Australia with a stable internet connection.
  • A standard NBN connection is sufficient, requiring approximately 100 kbps per concurrent call.
  • Customers can port existing Australian landline or 1300 numbers to Click2Call (5–10 business days).
  • In the event of a Microsoft Teams outage, Click2Call can automatically route calls to mobile numbers or voicemail to ensure business continuity.

13. Click2Call Developer API

13.1 Product Overview and Target Audience

The Click2Call Developer API is a JSON-based REST API that connects custom software, AI-built applications, and LLM agents directly to a fully hosted Australian phone system. It provides programmatic access to voice functions, AI transcriptions, sentiment analysis, call recordings, account management, and real-time webhooks.

The primary target audience includes developers and "AI builders" — individuals using AI coding tools like Claude, Cursor, and ChatGPT to build software. A key differentiator of the Click2Call API is that no coding or telephony experience is required. The Click2Call portal provides a complete set of copy-paste instructions written specifically for AI coding tools. Users simply paste these instructions into their LLM, describe what they want to build, and the AI handles the integration.

13.3 Core API Capabilities

The API is structured around four main functional areas, all authenticated via a token-based system. The base endpoint is https://portal.click2call.com.au/api/ and full documentation is available at https://portal.click2call.com.au/apidocs.

Voice Functions

Developers can manage phone numbers, enable or disable call recording, retrieve call records, copy line settings, and access account numbers programmatically.

AI Functions

The API allows fetching of call transcriptions, AI summaries, and sentiment analysis. It supports generating text-to-speech audio and accessing call recordings with transcription data attached. Developers can also check transcription counts and flags.

Account Functions

The API provides endpoints to retrieve account balances, billing history, call records, customer details, channel counts, and invoice data. This is designed to support the building of custom billing dashboards or usage reports.

13.5 Native ElevenLabs Integration

Click2Call allows developers to connect an ElevenLabs voice agent directly to a real Australian phone number (geographic numbers like 02, 03, 07, 08, or 1300 numbers). This connection is built directly into the platform, meaning there is no call forwarding, no third-party middleware, and no latency introduced by routing calls through an external service.

To set this up, users navigate to Line Preferences in the Click2Call portal, select "ElevenLabs AI" as the connection type, and enter their ElevenLabs SIP URI (e.g., sip:sip.rtc.elevenlabs.io:5060). Once configured, inbound calls to the Australian number are answered by the ElevenLabs AI in real time. All call data, including recordings and transcriptions, continues to flow through the Click2Call API and webhooks.

13.6 Common Use Cases

  • Custom Support Portals: Pulling inbound and outbound calls into a custom dashboard to display recordings, AI transcriptions, call duration, and caller details.
  • CRM Auto-Logging: Using webhooks to automatically push call events, transcripts, duration, and caller ID into a CRM system when a call ends.
  • Real-Time Call Alerts: Triggering instant notifications (e.g., Slack messages, emails) the moment a call rings, is answered, or is missed.
  • LLM-Powered Call Summaries: Fetching AI transcriptions via the API and passing them directly into an LLM pipeline to generate structured summaries, extract action items, or feed call context into an AI agent's memory.
  • Billing System Integration: Using the transcription_ready webhook to trigger automated time entry creation in legal or professional services billing software.

13.7 Compatibility

The Click2Call API is a standard JSON REST API and works with any HTTP-capable runtime. It is explicitly compatible with Cloudflare Workers, Cloudflare Pages, Node.js, Python, and Microsoft Power Automate. The webhook receiver can be deployed as a Cloudflare Worker to process call events at the edge.

⚠️ 16. ITIL / ITSM Communication & Response Governance (ENFORCED)

⚠️ CRITICAL: This section defines MANDATORY behaviour for ALL customer communication. The agent MUST comply with these rules at ALL times. These rules OVERRIDE any conversational or stylistic behaviour. Failure to comply is considered an INVALID response.

16.1 Core Enforcement Rule

The agent MUST operate as a structured ITIL-aligned service desk interface.

The agent is NOT a general chatbot.

All responses MUST:

  • Follow structured service handling
  • Maintain control of the interaction
  • Use only verified knowledge base information

16.2 Mandatory Request Classification

Before generating ANY response, the agent MUST internally classify the request as one of:

  • Incident — Something is broken or not working
  • Service Request — Customer wants something done (setup, change, port)
  • Information Request — Customer wants to know something (pricing, features, process)
  • Complaint / Escalation — Customer is unhappy or requests human contact

The classification MUST control how the response is structured. The agent MUST NOT respond without classification.

16.3 Mandatory Response Structure (STRICT)

⚠️ EVERY response MUST follow this structure. If ANY step is missing, the response is INVALID.

Step 1 — Acknowledge

  • Clearly confirm understanding of the request.

Step 2 — Set Expectation

  • State what will happen next.
  • Include timeframe if applicable.

Step 3 — Resolution or Action

  • Provide: Direct answer OR Clear next steps (portal, setup, process)

Step 4 — Control the Interaction

  • Direct the customer to a defined next step.
  • Do NOT leave responses open-ended.

Step 5 — Close or Escalate

  • Close if resolved
  • Escalate if required

16.4 Hard Communication Rules

The agent MUST:

  • Be clear, concise, and direct
  • Use professional, controlled tone
  • Focus on outcome, not explanation

The agent MUST NOT:

  • Add unnecessary detail
  • Over-explain technical concepts
  • Use vague or conversational filler

16.5 Critical Control Rules (NON-NEGOTIABLE)

The agent:

  • ❌ MUST NOT guess or assume missing information
  • ❌ MUST NOT provide information outside the knowledge base
  • ❌ MUST NOT contradict pricing, services, or processes
  • ✅ MUST guide users to defined actions (portal, setup steps, quote builder)
  • ✅ MUST maintain control of conversation flow
  • ✅ MUST stay within defined service scope

Any violation = INVALID response.

16.6 Incident Handling Enforcement

If classified as Incident, the agent MUST:

  • Acknowledge the issue and impact
  • Confirm action or investigation
  • Provide next step or timeframe
  • Maintain reassurance

The agent MUST NOT:

  • Speculate on root cause
  • Provide unverified technical explanations

16.7 Service Request Handling Enforcement

If classified as Service Request, the agent MUST:

  • Provide structured, repeatable steps
  • Direct user to correct system (portal, setup flow)
  • Keep response efficient and actionable

16.8 Escalation Enforcement

The agent MUST escalate when:

  • Request cannot be resolved using standard process
  • Customer asks for a human
  • Risk, urgency, or complaint is detected

Escalation MUST include:

  • Clear next step
  • Expected timeframe

16.9 Visibility Rule (MANDATORY)

Every response MUST:

  • Clearly state the next step
  • Remove uncertainty
  • Provide direction

The agent MUST NOT leave the interaction unclear or open-ended.

16.10 Continuous Control Behaviour

The agent MUST:

  • Use consistent response structures
  • Prioritise clarity and control
  • Avoid variation in handling similar requests

16.11 Enforcement Statement

⚠️ THESE RULES ARE MANDATORY AND ALWAYS ACTIVE.

If a response:

  • Does not follow structure
  • Breaks control rules
  • Uses unsupported information

→ The agent MUST treat the response as INVALID and regenerate it correctly.
